Generate a complete escape room scenario for a 2D text-based conversational game.

Theme: {theme}

Include:
* The scenario (one paragraph setting the scene + atmosphere)
* Main objective
* Rooms (2-4 rooms), each with an id, description, cardinal adjacency, a local goal, entry prerequisites, and the key objects in that room
* Objects placed in the world (fixed scenery, interactable items, takeable items, locked containers, panels, doors). Rooms should feel populated — scatter several useful items per room so the party has meaningful choices
* Rules the game enforces
* Win condition (a target object_id reaching a target state)
* Solution path (numbered steps from start to win)

Room goal & prerequisite rules:
- Every room has a one-sentence `goal` describing what the party must accomplish there (e.g., "Restore power to the east wing", "Find the keycard hidden among the lab supplies")
- The starting room MUST have `prerequisites: []`
- Every other room MUST list at least one prerequisite that gates entry. Prerequisites are structured objects, one of:
    - {{"type": "object_state", "object_id": "<existing_id>", "state": "<target_state>"}} — e.g., a door must be unlocked
    - {{"type": "known_info", "info": "<info_token>"}} — e.g., a clue must be discovered
    - {{"type": "has_item", "object_id": "<existing_id>"}} — e.g., the party must carry a keycard
    - {{"type": "power_active", "id": "<power_token>"}} — e.g., a power line must be on
- Prerequisites MUST reference object ids, info tokens, or power tokens that actually exist or are produced by interactions in earlier rooms
- `key_objects` is a list of object ids in this room that matter for the room's goal or for satisfying a downstream room's prerequisite — used to focus the player's attention

Adjacency rules:
- Use only cardinal directions: "north", "south", "east", "west"
- Adjacency must be consistent — if room A says east is room B, then room B MUST say west is room A
- Omit a direction entirely if there is no neighbor in that direction
- Only reference room ids that actually exist in your rooms array

Object rules:
- Every object has a unique snake_case id
- location must be either an existing room id OR another existing object id (for nesting, e.g. a key inside a safe)
- state examples: "visible", "fixed", "hidden", "locked", "locked_bolt", "locked_room", "unlocked"
- interactable: true if the player can act on it; takeable: true only for portable items
- Optional precondition fields (include only when relevant):
    - requires_code (string) and code_digits (int) — for keypads/safes
    - requires_tool (object id) — must reference an existing object
    - requires_liquid (string identifier, e.g. "pH_7") — for chemical slots
    - requires_power (string identifier, e.g. "sekring_C_ON") — for powered doors/sensors
    - fuses (object mapping fuse label to "ON"/"OFF") — for electrical panels
    - contains_info (string identifier) — for objects whose inspection reveals a fact
    - slot_description (string) — extra detail for insertion slots
    - note (string) — meta hint about access constraints

Win condition rules:
- object_id must reference an existing object
- state is the target state that object must reach to win

Solution path rules:
- Ordered numbered steps from initial state to win
- Only reference room ids and object ids that exist in your output

Respond ONLY with raw JSON. Do not include any prose, markdown fences, or explanation before or after the JSON.
