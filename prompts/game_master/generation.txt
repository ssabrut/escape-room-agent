Room goal & prerequisite rules (two-room mode):
- Each room has a one-sentence `goal` describing what the party must accomplish there.
- Each room has a `goal_completion`: a structured condition that is TRUE exactly when the room's goal is done. It must require non-trivial effort (a keypad, a tool, a fuse panel — never just walking in).
- Room A (start): `prerequisites: []`. Its `goal_completion` produces whatever Room B needs to enter (e.g., a door unlocked, a keycard taken, a clue learned).
- Room B (final): `prerequisites` MUST contain exactly Room A's `goal_completion` (verbatim — same `type`, `object_id`, `state`/`info`/`id`). Its `goal_completion` MUST equal the `win_condition`.
- `key_objects` is a list of object ids in this room that matter for its goal — used to focus the player's attention.
- Prerequisite / goal_completion shapes:
    - {{"type": "object_state", "object_id": "<existing_id>", "state": "<target_state>"}} — e.g., a door must be unlocked
    - {{"type": "known_info", "info": "<info_token>"}} — e.g., a clue must be discovered
    - {{"type": "has_item", "object_id": "<existing_id>"}} — e.g., the party must carry a keycard
    - {{"type": "power_active", "id": "<power_token>"}} — e.g., a power line must be on
- CRITICAL LINKAGE RULE: Room B's prerequisite MUST equal Room A's `goal_completion`. The only way to enter Room B is to finish Room A's puzzle. No shortcuts.

Adjacency rules:
- Use only cardinal directions: "north", "south", "east", "west"
- Room A and Room B MUST be cardinal neighbors. If A says east is B, then B MUST say west is A.
- No other adjacency entries — only the single edge between A and B.

Solvability rule (CRITICAL):
- Every code (`requires_code`) MUST be discoverable in the world. At least one object whose `contains_info` matches the code (or contains the code substring) must exist in a reachable location BEFORE the locked object. Same for liquids, powers, and tools — the means to satisfy a precondition must already be reachable when the precondition is needed.
- Do NOT invent codes or clues that only appear in the `solution_path`. The runtime ignores the solution path; players can only act on what is in `objects[]` and what `contains_info` reveals.

Object rules:
- Every object has a unique snake_case id
- location must be either an existing room id OR another existing object id (for nesting, e.g. a key inside a safe)
- state examples: "visible", "fixed", "hidden", "locked", "locked_bolt", "locked_room", "unlocked"
- interactable: true if the player can act on it; takeable: true only for portable items
- Optional precondition fields (include only when relevant):
    - requires_code (string) and code_digits (int) — for keypads/safes
    - requires_tool (object id) — must reference an existing object
    - requires_liquid (string identifier, e.g. "pH_7") — for chemical slots
    - requires_power (string identifier, e.g. "sekring_C_ON") — for powered doors/sensors

    - contains_info (string identifier) — for objects whose inspection reveals a fact
    - slot_description (string) — extra detail for insertion slots
    - note (string) — meta hint about access constraints

Win condition rules:
- object_id must reference an existing object
- state is the target state that object must reach to win

Solution path rules:
- Ordered numbered steps from initial state to win
- Only reference room ids and object ids that exist in your output

Respond ONLY with raw JSON. Do not include any prose, markdown fences, or explanation before or after the JSON.
